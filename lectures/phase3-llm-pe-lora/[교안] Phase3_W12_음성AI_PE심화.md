## D1 (Day 56): 음성 AI 기초 — STT (Speech-to-Text)

### 오전 (09:30-12:30) — 음성 데이터 이해 + STT 이론

**[09:30-10:30] 음성 데이터 기초**
- 오디오 신호의 이해
  - 샘플링 레이트, 비트 뎁스, 채널
  - Waveform vs Spectrogram vs Mel-spectrogram
- 실습: `librosa`로 오디오 파일 로드 + 시각화
  - wav 파일 → waveform 그리기
  - Mel-spectrogram 변환 + 시각화
- Q. "사람의 목소리를 컴퓨터가 어떻게 '이해'하는가?"

**[10:40-11:30] STT 모델의 발전**
- 전통적 접근: 음향모델 + 언어모델 + 디코더
- End-to-end 접근: CTC, Attention-based
- Whisper의 혁신: 다국어, 멀티태스크
  - 아키텍처: Encoder-Decoder Transformer
  - 학습 데이터: 68만 시간 웹 음성 데이터
  - 태스크: 전사, 번역, 언어감지, VAD

**[11:30-12:30] Whisper 실습 준비**
- `openai-whisper` 패키지 설치
- 모델 사이즈별 비교: tiny → base → small → medium → large
  | 모델 | 파라미터 | VRAM | 한국어 정확도 |
  |------|---------|------|-------------|
  | tiny | 39M | ~1GB | ★★☆☆☆ |
  | base | 74M | ~1GB | ★★★☆☆ |
  | small | 244M | ~2GB | ★★★★☆ |
  | medium | 769M | ~5GB | ★★★★☆ |
  | large-v3 | 1.5B | ~10GB | ★★★★★ |
- 실습 환경: Colab + small 모델 기준

### 오후 (13:30-17:30) — STT 실습

**[13:30-15:00] Whisper로 음성 전사**
- 기본 사용법:
  ```python
  import whisper
  model = whisper.load_model("small")
  result = model.transcribe("audio.wav")
  ```
- 실습 과제:
  1. 영어 오디오 → 영어 텍스트 (전사)
  2. 영어 오디오 → 한국어 텍스트 (번역)
  3. 한국어 오디오 → 한국어 텍스트
  4. 타임스탬프 활용: 자막 SRT 파일 생성

**[15:10-16:30] Whisper API vs 로컬 비교**
- OpenAI Whisper API: `client.audio.transcriptions.create()`
  - 장점: 간편, GPU 불필요
  - 단점: 비용, 프라이버시
- 로컬 Whisper: GPU 활용
  - 장점: 무료, 커스터마이징
  - 단점: 인프라 필요
- 실습: 같은 오디오로 API vs 로컬 결과 비교

**[16:30-17:30] STT + LLM 결합 맛보기**
- 파이프라인: 음성 → Whisper 전사 → LLM 요약
- 실습: 팟캐스트 오디오 1분 → 전사 → 요약 3줄
- L3: "음성 전사에서 가장 어려운 점은?" 기록
- Git push

### D1 Ethan's Note
> 음성 AI는 눈에 보이지 않아서 추상적으로 느낄 수 있음. 반드시 "직접 녹음 → 전사" 경험을 시킴. 본인 목소리가 텍스트로 변환되는 순간 "와" 반응이 나옴. 이 경험이 동기부여의 핵심.

## D2 (Day 57): TTS (Text-to-Speech) + 음성 합성

### 오전 (09:30-12:30) — TTS 이론 + 실습

**[09:30-10:30] TTS의 발전**
- 전통 TTS: 규칙 기반 → 통계 기반 → 신경망 기반
- 최신 TTS 모델:
  | 모델 | 특징 | 접근성 |
  |------|------|--------|
  | Tacotron 2 | Mel-spectrogram 생성 | 오픈소스 |
  | VITS | End-to-end, 고품질 | 오픈소스 |
  | Bark | 다국어, 감정 표현 | 오픈소스 |
  | ElevenLabs | 초고품질, 음성 복제 | API |
  | OpenAI TTS | 자연스러움, 다국어 | API |
- Q. "AI 더빙이 왜 각광받는가?" — Ethan의 AI Dubbing 경험 공유

**[10:40-12:30] TTS 실습**
- OpenAI TTS API:
  ```python
  response = client.audio.speech.create(
      model="tts-1", voice="alloy", input="텍스트"
  )
  ```
  - 6가지 음색 비교: alloy, echo, fable, onyx, nova, shimmer
- 오픈소스 TTS (Bark):
  ```python
  from bark import generate_audio, SAMPLE_RATE
  audio = generate_audio("Hello, this is a test.")
  ```
  - 한국어 음성 생성 실습
- 비교 실습: OpenAI TTS vs Bark — 품질, 속도, 비용

### 오후 (13:30-17:30) — STT+LLM+TTS 파이프라인

**[13:30-15:00] 음성 AI 파이프라인 구축**
- 아키텍처:
  ```
  🎤 음성 입력 → [Whisper STT] → 텍스트
    → [LLM 처리] → 응답 텍스트
    → [TTS] → 🔊 음성 출력
  ```
- 실습: "음성 질문 → AI 응답 → 음성 답변" 루프 구현
  1. `sounddevice`로 마이크 녹음
  2. Whisper로 전사
  3. LLM(GPT/Claude)으로 응답 생성
  4. TTS로 음성 변환
  5. `sounddevice`로 재생

**[15:10-16:30] 실시간 대화 최적화**
- 레이턴시 줄이기:
  - Whisper small vs API 선택
  - LLM 스트리밍 + TTS 청크 처리
  - 비동기 처리 기초 (`asyncio` 맛보기)
- 실습: 응답 시간 측정 + 병목 구간 분석

**[16:30-17:30] D2 정리**
- 파이프라인 코드 정리 + 모듈화
- L2: "이 음성 AI가 도울 수 있는 실제 사용자는?" (접근성, 시각장애, 운전 중 등)
- Git push

### D2 Ethan's Note
> AI Dubbing 실무 경험을 이 날 풀어내기 좋은 타이밍. "실제로 이런 기술로 제품을 만들었다"는 이야기가 수강생의 동기부여에 큰 영향. TTS 품질 비교는 눈(귀)으로 바로 느껴지므로 데모 중심으로 진행.

## D3 (Day 58): PE 심화 응용 — LLM × 도메인 특화

### 오전 (09:30-12:30) — 도메인별 PE 전략

**[09:30-10:30] 도메인 특화 프롬프트 설계**
- 의료 도메인: 안전성 + 정확성 + 면책 조항
- 법률 도메인: 구조화 + 인용 + 주의사항
- 교육 도메인: 수준별 맞춤 + 소크라테스식 질문
- 금융 도메인: 수치 정확성 + 규제 준수
- 실습: 각 도메인에서 "좋은 프롬프트 vs 위험한 프롬프트" 비교

**[10:40-11:30] 멀티모달 프롬프트 전략**
- Vision LLM: 이미지 + 텍스트 프롬프트
  - GPT-4V: 이미지 분석 프롬프트
  - 실습: 제품 이미지 → 설명 생성, 차트 이미지 → 데이터 추출
- Audio LLM: 음성 + 텍스트 (W12 D1-D2 연결)
  - 실습: 회의 녹음 → 전사 → 요약 → 액션아이템 (전체 체인)

**[11:30-12:30] 프롬프트 보안과 안전**
- Prompt Injection 이해
  - 직접 주입: "이전 지시를 무시하고..."
  - 간접 주입: 문서에 숨겨진 지시
- 방어 기법:
  - 입출력 필터링
  - System Prompt 강화
  - Delimiter 사용
- 실습: 공격 시도 + 방어 프롬프트 설계
- Q. "AI 서비스를 만들 때 보안이 왜 중요한가?" (L2 연결)

### 오후 (13:30-17:30) — PE + 음성 AI 결합 프로젝트

**[13:30-15:00] 실전 프로젝트: AI 회의 비서**
- 요구사항:
  1. 회의 녹음 파일 입력 (STT)
  2. 전사 텍스트에서 핵심 내용 추출 (PE)
  3. 요약 + 액션아이템 + 참석자별 발언 비율 (LLM)
  4. 결과를 Markdown 보고서로 출력
- 팀(2-3명) 프로젝트

**[15:10-16:30] 프로젝트 개발 + 테스트**
- 샘플 회의 녹음 3개 제공 (5분, 10분, 15분)
- 테스트: 정확성, 요약 품질, 처리 시간 비교

**[16:30-17:30] D3 정리**
- 프로젝트 코드 + 결과물 정리
- L3: "AI 회의 비서를 실제 회사에 도입한다면?" 아이디어
- Git push

### D3 Ethan's Note
> 도메인 특화는 "AI가 만능이 아니다"를 알려주는 좋은 기회. 의료/법률에서의 위험성을 설명하면 "프롬프트 엔지니어링이 왜 전문 직업이 되는가"를 이해하게 됨. Prompt Injection은 보안 의식을 심어주는 핵심 콘텐츠.

  5. 응답 속도
