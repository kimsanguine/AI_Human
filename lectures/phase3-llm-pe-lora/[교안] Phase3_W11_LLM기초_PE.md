  - 자기회귀(Autoregressive) 모델: 다음 토큰 예측

  - 실습: `tiktoken`으로 한국어 vs 영어 토큰 수 비교
  - Q. "왜 한국어가 더 많은 토큰을 쓸까?" → 비용과 성능 영향

**[11:30-12:30] LLM 생태계 지도**

  | 장점 | 커스터마이징, 비용 통제 | 성능, 편의성 |
  | 단점 | 인프라 필요, 성능 격차 | 의존성, 비용 |
- HuggingFace 생태계 투어
  - Model Hub, Spaces, Datasets
  - 실습: HuggingFace에서 모델 카드 읽는 법

## D2 (Day 52): Prompt Engineering 기초

### 오전 (09:30-12:30) — PE 핵심 기법

**[09:30-10:30] 왜 프롬프트가 중요한가**
- Q. "같은 모델인데 왜 결과가 천차만별인가?"
- 프롬프트 = 모델의 행동을 정의하는 인터페이스
- 채용시장 현황: PE가 Tier 1 스킬인 이유 (85%+ 채용공고 언급)
- System Prompt vs User Prompt vs Assistant Prompt 역할

**[10:40-11:30] 핵심 PE 기법 4가지**
1. **Zero-shot**: 지시만으로 태스크 수행
   - 실습: "이 리뷰의 감성을 분석해줘"
2. **Few-shot**: 예시를 제공하여 패턴 학습
   - 실습: 3개 예시 + 새 입력 → 분류
3. **Chain of Thought (CoT)**: "단계별로 생각해봐"
   - 실습: 수학 문제 → CoT 유무 비교
4. **Role Prompting**: "너는 ~전문가야"
   - 실습: 같은 질문, 다른 역할 → 답변 비교

**[11:30-12:30] 프롬프트 구조 설계**
- Ethan의 프롬프트 템플릿:
  ```
  # Your role: {역할}
  # Objectives: {목표}
  # Reasoning strategy: {추론 방식}
  # Context and data: {맥락}
  # Output format: {출력 형식}
  ```
- 실습: 이 템플릿으로 "경쟁사 분석 프롬프트" 만들기
- Q. "좋은 프롬프트와 나쁜 프롬프트의 차이는?"

### 오후 (13:30-17:30) — PE 실습 워크숍

**[13:30-15:00] 실전 프롬프트 설계 챌린지**
- 태스크 5개를 주고, 각각 최적의 프롬프트 설계:
  1. 고객 리뷰 감성분석 + 카테고리 분류
  2. 회의록 요약 → 액션아이템 추출
  3. 기술 문서 → 비개발자용 설명 변환
  4. 이력서 → 강점/약점 분석
  5. 코드 리뷰 → 버그 및 개선점 제안
- 각 태스크별 A/B 테스트: 두 가지 프롬프트로 비교

**[15:10-16:30] 프롬프트 평가 프레임워크**
- 좋은 프롬프트의 기준:
  | 기준 | 설명 | 측정법 |
  |------|------|--------|
  | 정확성 | 원하는 답을 내는가 | 정답률 |
  | 일관성 | 여러 번 돌려도 같은가 | 분산 |
  | 구조화 | 출력 형식이 일정한가 | 파싱 성공률 |
  | 효율성 | 토큰을 낭비하지 않는가 | 토큰 수 |
- 실습: 위 5개 태스크 중 1개를 골라 10번 반복 → 일관성 평가

**[16:30-17:30] PE 포트폴리오 시작 + Git Push**
- 오늘 만든 프롬프트를 정리해서 PE-portfolio.md 작성
- L2: "이 프롬프트가 해결하는 사용자 문제는?"을 각 프롬프트에 추가
- GitHub에 push

### D2 Ethan's Note
> PE는 "기술"이 아니라 "사고방식"임을 강조. PM 경험에서 "요구사항 정의"와 "프롬프트 설계"가 같은 근육이라는 점을 연결. 수강생이 "이거 내가 잘할 수 있겠다"는 자신감을 갖게 하는 게 이 날의 목표.

## D3 (Day 53): Prompt Engineering 심화

### 오전 (09:30-12:30) — 고급 PE 기법

**[09:30-10:30] 구조화된 출력 (Structured Output)**
- JSON 모드: `response_format={"type": "json_object"}`
- 실습: LLM에게 JSON으로 응답받기
  - 입력: 제품 리뷰 텍스트
  - 출력: `{"sentiment": "positive", "score": 0.85, "keywords": [...]}`
- Pydantic으로 출력 검증하기
- Q. "왜 구조화된 출력이 중요한가?" → 후속 시스템 연동

**[10:40-11:30] 고급 기법들**
- Self-consistency: 여러 번 추론 → 다수결
- Tree of Thought: 분기점에서 여러 경로 탐색
- ReAct: Reasoning + Acting 패턴 (W16 Agent의 기초)
- 실습: CoT vs Self-consistency 비교 실험

**[11:30-12:30] 프롬프트 체이닝**
- 복잡한 태스크를 여러 프롬프트로 분해
- 실습 시나리오: "뉴스 기사 → 요약 → 감성분석 → 보고서 생성"
  - Step 1: 기사 요약 프롬프트
  - Step 2: 요약 결과 감성분석 프롬프트
  - Step 3: 분석 결과 보고서 생성 프롬프트
- Q. "한 번에 다 시키면 안 되나?" → 품질 vs 편의 트레이드오프

### 오후 (13:30-17:30) — 실전 프로젝트

**[13:30-15:00] 미니 프로젝트: AI 고객 서비스 봇 프롬프트 설계**
- 시나리오: 이커머스 고객 문의 처리 봇
- 요구사항:
  1. 문의 유형 분류 (배송/환불/제품문의/기타)
  2. 유형별 응답 생성
  3. 에스컬레이션 판단 (사람에게 넘길지)
- 팀(2-3명) 단위로 프롬프트 시스템 설계
- System Prompt + 유형별 User Prompt 템플릿

**[15:10-16:30] 프롬프트 시스템 테스트 + 발표**
- 각 팀이 만든 프롬프트 시스템을 10개 테스트 케이스로 평가
- 팀별 5분 발표: 설계 의도 + 테스트 결과
- 동료 피드백

**[16:30-17:30] D3 정리**
- PE 치트시트 작성 (본인만의 프롬프트 가이드)
- L3: "프롬프트 설계 시 가장 중요한 3가지 원칙" 정리
- Git push

### D3 Ethan's Note
> 심화 기법은 "외우는 것"이 아니라 "언제 쓰는지"를 아는 것. ReAct 패턴은 W16 Agent와 직접 연결되므로 "이건 나중에 Agent 만들 때 다시 나온다"고 예고. 프롬프트 체이닝은 W14 LangChain의 Chain 개념과 연결됨을 암시.
